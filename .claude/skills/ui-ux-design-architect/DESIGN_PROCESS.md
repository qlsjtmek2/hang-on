## 요구사항 분석 심화

### 요구사항 분류

**기능적 요구사항** (Functional Requirements):

- 사용자가 수행할 수 있는 작업
- 시스템이 제공해야 하는 기능
- 입력과 출력

**비기능적 요구사항** (Non-Functional Requirements):

- 성능 (응답 시간, 처리량)
- 보안 (데이터 암호화, 인증)
- 접근성 (WCAG 준수)
- 확장성 (동시 사용자 수)

### MoSCoW 우선순위

| 우선순위        | 의미 | 설명                 |
| --------------- | ---- | -------------------- |
| **Must have**   | P0   | MVP 필수 기능        |
| **Should have** | P1   | 중요하지만 연기 가능 |
| **Could have**  | P2   | 있으면 좋음          |
| **Won't have**  | -    | 이번 릴리스에서 제외 |

### 요구사항 추출 질문

1. **사용자 목표**: 사용자가 이 앱으로 무엇을 달성하려 하나?
2. **핵심 가치**: 이 앱의 차별점은 무엇인가?
3. **제약 조건**: 기술적/비즈니스적 제약이 있나?
4. **성공 지표**: 어떻게 성공을 측정할 것인가?

## 사용자 리서치 방법론

### 정성적 리서치

**1. 사용자 인터뷰**

- 목적: 사용자의 needs, pain points 파악
- 방법: 1:1 심층 인터뷰 (30-60분)
- 질문 예시:
  - "현재 감정을 기록할 때 어떤 방법을 사용하나요?"
  - "익명으로 감정을 공유할 때 가장 걱정되는 점은?"

**2. 관찰 조사 (Contextual Inquiry)**

- 목적: 실제 사용 환경에서 행동 관찰
- 방법: 사용자의 자연스러운 환경에서 관찰

**3. 포커스 그룹**

- 목적: 여러 사용자의 의견 수렴
- 방법: 6-8명 그룹 토론 (60-90분)

### 정량적 리서치

**1. 설문 조사**

- 목적: 대규모 데이터 수집
- 도구: Google Forms, Typeform

**2. 분석 (Analytics)**

- 목적: 실제 사용 패턴 분석
- 도구: Google Analytics, Mixpanel

```
**페르소나 #1: [이름]**

**인구통계**:
- 나이: [나이]
- 직업: [직업]
- 거주지: [도시]

**성격 특성**:
- [특성 1]
- [특성 2]
- [특성 3]

**목표**:
- [주요 목표 1]
- [주요 목표 2]

**Pain Points**:
- [문제점 1]
- [문제점 2]

**기술 숙련도**:
- 스마트폰 사용: [초급/중급/고급]
- 앱 사용 빈도: [일일 X시간]

**선호 채널**:
- [SNS, 메신저 등]

**한 줄 요약**:
"[페르소나를 한 문장으로 요약]"
```

### 예시 (Hang On 프로젝트)

```
**페르소나 #1: 감정 표현에 서툰 직장인 지우**

**인구통계**:
- 나이: 28세
- 직업: 마케터
- 거주지: 서울

**성격 특성**:
- 내향적
- 감정 표현에 서툼
- 완벽주의 성향

**목표**:
- 감정을 솔직하게 표현하고 싶음
- 타인의 판단 없이 위로받고 싶음

**Pain Points**:
- 주변 사람들에게 속마음을 털어놓기 어려움
- SNS는 너무 공개적이라 부담스러움
- 일기는 외로운 느낌

**기술 숙련도**:
- 스마트폰 사용: 고급
- 앱 사용 빈도: 일일 3-4시간

**선호 채널**:
- 카카오톡, 인스타그램 (DM 위주)

**한 줄 요약**:
"익명으로 안전하게 감정을 표현하고 공감받고 싶은 내향적 직장인"
```

## 사용자 여정 매핑 심화

### 사용자 여정 맵(Customer Journey Map) 구성 요소

1. **단계(Stages)**: 사용자가 거치는 주요 단계
2. **행동(Actions)**: 각 단계에서 하는 행동
3. **생각(Thoughts)**: 각 단계에서의 생각
4. **감정(Emotions)**: 감정 변화 (긍정/부정)
5. **Touchpoints**: 상호작용하는 화면/기능
6. **Pain Points**: 불편한 점
7. **Opportunities**: 개선 기회

### 템플릿

| 단계    | 행동        | 생각               | 감정   | Touchpoint | Pain Point     | Opportunity |
| ------- | ----------- | ------------------ | ------ | ---------- | -------------- | ----------- |
| 1. 발견 | 앱 다운로드 | "이게 뭐지?"       | 궁금함 | 앱스토어   | 설명 부족      | 온보딩 개선 |
| 2. 가입 | 이메일 입력 | "빨리 써보고 싶다" | 기대감 | 가입 화면  | 입력 필드 많음 | 소셜 로그인 |
| ...     | ...         | ...                | ...    | ...        | ...            | ...         |

### Emotional Timeline

사용자 여정에서 감정 변화를 그래프로 시각화:

```
긍정 (+5) ┤      ╱╲
          │     ╱  ╲
중립 (0)  ┼────╱────╲─────
          │           ╲  ╱
부정 (-5) ┤            ╲╱
          └─────────────────
           발견 가입 첫사용 습관화
```

## 정보 구조(IA) 설계

### 정보 구조 패턴

**1. Hierarchy (계층 구조)**

- 트리 형태: 상위 → 하위
- 예: 설정 > 계정 > 비밀번호 변경

**2. Sequential (순차 구조)**

- 단계별 진행: 1단계 → 2단계 → 3단계
- 예: 회원가입, 온보딩

**3. Matrix (매트릭스)**

- 다차원 탐색: 태그, 필터
- 예: 쇼핑몰 카테고리

**4. Hub and Spoke (허브-스포크)**

- 중앙 허브 → 개별 기능
- 예: 홈 화면 → 각 메뉴

### Hang On 프로젝트 IA

```
[앱 루트]
├─ [메인 화면] (Hub)
│  ├─ 내 기록 목록
│  ├─ [털어놓기] → [감정 선택] → [글쓰기] (Sequential)
│  └─ [누군가와 함께] → [타인 기록 피드]
├─ [설정] (Hierarchy)
│  ├─ 계정 설정
│  ├─ 알림 설정
│  ├─ 언어 설정
│  └─ 계정 삭제
└─ [인증]
   ├─ 회원가입
   └─ 로그인
```

## 와이어프레임 작성

### Low-Fidelity vs High-Fidelity

| 특성   | Low-Fidelity       | High-Fidelity     |
| ------ | ------------------ | ----------------- |
| 디테일 | 낮음 (스케치)      | 높음 (거의 완성)  |
| 시간   | 빠름 (분 단위)     | 느림 (시간 단위)  |
| 도구   | 종이, 화이트보드   | Figma, Sketch     |
| 목적   | 초기 아이디어 검증 | 개발 전 최종 확인 |

### 와이어프레임 작성 원칙

1. **회색조 사용**: 색상은 나중에 (구조에 집중)
2. **실제 콘텐츠**: "Lorem ipsum" 대신 실제 텍스트 사용
3. **일관된 간격**: 8pt Grid 유지
4. **주석 추가**: 인터랙션, 상태 변화 설명

### 텍스트 기반 와이어프레임 예시

## 프로토타이핑

### 프로토타입 종류

**1. Paper Prototype**

- 종이에 그린 화면을 손으로 전환
- 매우 빠른 검증
- 도구: 종이, 펜

**2. Clickable Prototype**

- 클릭 가능한 디지털 프로토타입
- 실제 사용 흐름 테스트
- 도구: Figma, Adobe XD, Framer

**3. Interactive Prototype**

- 애니메이션 포함
- 실제 앱과 유사한 경험
- 도구: Framer, ProtoPie

### Figma 프로토타이핑 체크리스트

- [ ] 모든 주요 화면 연결
- [ ] Happy Path 플로우 완성
- [ ] 뒤로가기/닫기 동작 추가
- [ ] 에러 상태 화면 포함
- [ ] 로딩 상태 표현
- [ ] 트랜지션 효과 적용 (200-300ms)

## 사용성 테스트

### 테스트 계획

**목표 설정**:

- "사용자가 5분 내 첫 기록을 작성할 수 있는가?"
- "공유 옵션 선택 시 혼란스러워하지 않는가?"

**참가자**:

- 5-8명 (페르소나와 일치)
- 앱 미사용자

**태스크**:

1. 회원가입 후 첫 기록 작성
2. 타인의 기록에 공감하기
3. 과거 기록 찾아서 수정하기

### 테스트 진행

**1. 준비**:

- 프로토타입 또는 베타 앱
- 녹화 장비 (화면 녹화 + 카메라)
- 태스크 시나리오 문서

**2. Think Aloud (생각 말하기)**:

- 참가자에게 생각을 소리 내어 말하도록 요청
- "지금 무엇을 하려고 하시나요?"
- "이 버튼이 무슨 기능일 것 같나요?"

**3. 관찰 지표**:

- 태스크 완료 여부
- 완료 시간
- 오류 횟수 (잘못된 버튼 클릭 등)
- 주저하는 지점 (hesitation)

**4. 사후 인터뷰**:

- "가장 어려웠던 부분은?"
- "가장 좋았던 부분은?"
- "개선하고 싶은 부분은?"

### 테스트 결과 정리

**발견 사항(Findings) 템플릿**:

| 이슈                  | 심각도 | 빈도  | 개선안           |
| --------------------- | ------ | ----- | ---------------- |
| 공유 옵션이 헷갈림    | 높음   | 5/5명 | 설명 텍스트 추가 |
| 뒤로가기 버튼 못 찾음 | 중간   | 2/5명 | 헤더 고정        |
| ...                   | ...    | ...   | ...              |

**심각도**:

- **Critical**: 태스크 실패
- **High**: 많은 시간 소요
- **Medium**: 약간 불편
- **Low**: 사소한 개선

## Design Iteration (반복 개선)

디자인은 한 번에 완성되지 않습니다. 다음 사이클을 반복하세요:

```
1. Design → 2. Prototype → 3. Test → 4. Analyze → 1. Re-design
```

**개선 우선순위**:

1. Critical 이슈 먼저 해결
2. High 이슈 해결
3. Quick Win (쉽게 개선 가능한 것)
4. Medium/Low 이슈
